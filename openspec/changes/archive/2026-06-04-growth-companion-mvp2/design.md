## Context

Growth Companion MVP 已完成：FastAPI 后端 + 三层记忆 + JPAF 八维人格演化 + 单 HTML 前端 + 流式聊天。MVP2 在此基础上新增回顾和管理能力，全部在前端单 HTML 中实现，后端新增 API 端点。

## Goals / Non-Goals

**Goals:**
- 用户能回顾每周/每月/人生阶段的成长轨迹
- 用户能管理自己的日记（查看、补生成、重新生成）
- 用户能编辑核心记忆、管理事件、查看人格权重变化
- 所有新功能在前端有对应的 UI 入口

**Non-Goals:**
- 不做前端框架迁移（仍是单 HTML）
- 不做多用户系统
- 不做数据导出
- 不做通知/推送（周报需要用户或系统手动触发）

## Decisions

### D1: 周报和人生章节存储为 Markdown 文件

**选择**：与日记一致，存为 `data/{user_id}/summaries/week-YYYY-WNN.md` 和 `data/{user_id}/chapters/YYYY-标题.md`。

**替代方案**：存 SQLite。

**理由**：这些是生成物而非结构化数据，文件更直观，可直接查看和编辑。

### D2: 人格权重历史存 SQLite

**选择**：新增 `personality_snapshots` 表，每次 Reflection 后插入一条记录（权重 JSON + 时间戳）。

**替代方案**：存 JSON 文件。

**理由**：需要按时间范围查询和绘图，结构化查询更方便。

### D3: 周报触发方式

**选择**：提供 API 端点手动触发生成。前端在日记页面显示"生成周报"按钮。

**替代方案**：定时任务自动生成。

**理由**：MVP 阶段没有后台调度器。手动触发更简单，用户可在任意时间生成任意周的周报。

### D4: 前端新增两个侧边栏入口

**选择**：侧边栏新增"📖 日记"和"🧠 记忆"（原有记忆页面拆分为日记管理和记忆管理）。

**理由**：日记和记忆是不同的使用场景。日记是回顾，记忆是管理。

### D5: 人格历史可视化用简单柱状图

**选择**：纯 HTML/CSS 柱状图，不用图表库。

**理由**：单 HTML 方案，不引入额外依赖。8 个维度的变化趋势用颜色深浅和高度即可表达。

## Risks / Trade-offs

**[周报质量依赖事件数量]** → 如果当周事件少，周报会空洞。生成时检查事件数量，不足 3 条时提示"本周对话较少，总结可能不完整"。

**[重新生成会覆盖]** → 重新生成日记/周报前提示确认，或保留旧版本。

**[人格快照数据量]** → 单用户场景，每周约 1 条快照，年增长可忽略。不做清理。

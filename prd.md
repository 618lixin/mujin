# AI 智能日记系统 PRD

基于「降低输入成本 + 提高输出价值」的 AI 辅助日记管理产品

> **文档版本**: v0.3.0（2026-05-23 更新）
> **状态**: MVP + MVP2 已实现，v0.3 定位重构

------

# 1. 产品定位

## 产品名称（暂定）

- Growth Companion
- Echo
- 余声
- 长谈
- Seed

------

## 产品愿景

**两个核心点**：

1. **降低输入成本** — 不写日记，聊天就行。随口吐槽、随口分享，AI 自动记录。
2. **提高输出价值** — 孤立的日记被 AI 串联、提取、分析、对比，变成有价值的成长档案。

它不是：

- 情感陪伴型 AI（不是心理倾听者）
- 普通聊天机器人（不是一次性对话）
- 传统日记工具（不需要用户主动写，记完也不会孤立）

而是：

> "AI 辅助的智能日记系统 — 对话即记录，AI 帮你串联、提取、分析、对比"

------

## 核心公式

```
碎片对话 → AI 自动记录 → AI 串联分析 → 有价值的成长档案
  (降成本)      (降成本)      (提价值)       (提价值)
```

------

## 功能取舍标准

所有功能用这两个标准衡量：

- 它降低了输入成本吗？
- 它提高了输出价值吗？

两个都不贡献的功能，不做。

------

# 2. 核心理念

------

## 2.1 对话即日记（降低输入成本）

传统日记的问题：

- 需要坐下来"写"（高门槛）
- 需要组织语言（费精力）
- 写完就不看了（低价值）

解决方案：

- 用户只需要聊天，不需要"写"
- 随口吐槽、随口分享，AI 自动生成日记
- 不需要组织语言，对话本身就是素材

------

## 2.2 AI 串联日记（提高输出价值）

传统日记的致命问题：**每篇都是孤立的**。

3月5日写了一篇，3月20日写了一篇，永远不会再有交集。

AI 做的事：

- **串联** — 把零散日记按主题/项目串联起来（"考研"从开始到结果）
- **提取** — 从日记中提取关键事件、情绪、行为模式
- **分析** — 发现趋势、模式、变化（"你最近比三个月前更果断了"）
- **对比** — 时间维度对比（"9月的你和3月完全不同"）

输出产物：

- 项目档案（跨日记的主题聚合）
- 周报/月报（时间维度的总结）
- 成长观察（变化趋势的定性描述）
- 人生章节（重要阶段的叙事）

系统分为五层：

```text

│ 对话层 Chat Layer │  ← SSE 流式 + prompt 组装（自然朋友风格 + 记忆注入）

┌────────────────────┐

└────────┬───────────┘
         ↓
┌────────────────────┐

         ↓

│ 沉淀生成器 │  ← 日记/周报/章节 + 项目档案/成长线
└────────────────────┘
```

# 7. Memory System（核心）

这是整个产品最重要部分。

设计灵感来自 Hermes Agent 的记忆架构：**有界、策展、始终激活**。

核心原则：

- 记忆不是数据库查询，而是"一直在脑子里"
- 硬上限逼迫策展——只留最重要的
- 不用向量数据库，SQLite + 文件系统足够 MVP
- 事件会遗忘——只有反复被想起的重要事件才能长期保留

------

## 7.1 三层记忆架构

------

### Layer 1：核心记忆（Core Memory）—— 始终在 prompt 中 ✅ 已实现

类似 Hermes 的 MEMORY.md + USER.md。

硬限 ~2000 字符，每轮对话注入 system prompt，AI 始终"记得"。

分为两个文件：

#### user_profile.md（用户画像，~1200 字符）

保存：

- 用户基本情况
- 沟通偏好
- 当前关注点
- 近期状态

例如：

```text
用户基本情况：互联网产品经理，在考虑转独立开发
沟通偏好：随和，不喜欢被说教
当前关注：职业方向、个人项目
近期状态：在纠结是否辞职
```

#### companion_notes.md（AI 笔记，~800 字符）

AI 的日记生成上下文笔记，帮助生成更贴合用户的日记：

- 用户当前关注点
- 对话模式观察（什么话题让用户兴奋/沉默）
- 日记生成偏好（用户喜欢什么风格）
- 重要上下文（跨会话的连贯性）

例如：

```text
用户最近在纠结要不要辞职做独立开发
之前聊过两次这个话题，态度从犹豫到倾向行动
聊技术细节时很兴奋，可以多追问
上次提到想做个日记 app，可能是很好的切入点
```

**容量管理**：

- 使用率 > 80% → 标记接近上限，提示 AI 压缩
- 超过 100% → 拒绝写入
- 写入操作：add（追加）、replace（子串替换）、remove（子串删除）

> **待改进**：当前容量管理有警告标记，但缺少 AI 主动合并/压缩的自动化逻辑。

------

### Layer 2：事件记忆（Event Memory）—— SQLite 结构化存储 ✅ 已实现

不注入 prompt，按需检索。

保存重要事件，带结构化元数据：

```sql
CREATE TABLE events (
  id TEXT PRIMARY KEY,
  content TEXT NOT NULL,
  emotions TEXT,          -- JSON array: ["sadness", "anger"]
  importance REAL,        -- 0.0 ~ 1.0
  event_type TEXT,        -- "conflict" | "milestone" | "emotion" | "decision"
  strength REAL,          -- 遗忘曲线当前强度 (0.0 ~ 1.0)
  stability REAL,         -- 记忆稳定天数
  recall_count INTEGER,   -- 被回忆的次数
  last_recalled_at TEXT,  -- 上次被回忆的时间
  created_at DATETIME,
  updated_at DATETIME
);
```

触发写入的条件：

- AI 识别到情绪剧烈波动（如崩溃、愤怒）
- 用户提到重要事件（如分手、面试、考试）
- AI 发现行为模式变化（如"第一次主动表达边界"）

检索方式：

- 按 importance 排序取最近 N 条
- 按 event_type 筛选
- 按时间范围查询
- 按 min_strength 过滤（已遗忘的不返回）

**遗忘曲线**（v0.2 新增）：

事件不是永久保留的。每条事件有 `strength`（记忆强度）和 `stability`（稳定天数）：

```
strength = e^(-elapsed_days / stability)
```

- 初始 stability = 基础稳定天数 × (0.5 + importance)
- 被回忆时 stability 增加 50%
- strength < 0.05 的事件可被清理
- 心跳循环每 30 分钟自动执行衰减和清理

------

### Layer 3：成长叙事（Narrative Artifacts）—— 生成物 ✅ 已实现

不是实时记忆，是定期生成的独立文件。

不注入 prompt，用户可查看，AI 生成周报/月报时可读取。

包括：

- 每日自动日记（`diaries/YYYY-MM-DD.md`）
- 每周成长总结（`summaries/week-YYYY-WNN.md`）
- 人生章节（`chapters/标题.md`）

这些是"产物"而非"记忆"——它们帮助用户看到自己的变化，
但不直接参与对话循环。

# 8. 成长观察（用户变化可视化）

------

## 8.1 定性观察 ✅ 待实现（v0.3 重构）

用自然语言描述用户的变化，而不是虚假的精确数字。

### 设计理念

人的成长无法用算法精确量化。±0.02 的数字看起来严谨，
但实际上只是 LLM 的主观猜测。不如用诚实的文字描述变化。

### 数据结构

```sql
CREATE TABLE observations (
  id TEXT PRIMARY KEY,
  date TEXT NOT NULL,           -- 观察日期
  content TEXT NOT NULL,        -- 观察内容，如"你开始更多表达自己的感受"
  category TEXT,                -- "emotion" | "behavior" | "relationship" | "value" | "growth"
  source TEXT,                  -- "reflection" | "event"
  created_at DATETIME
);
```

### 用户可见

前端展示为时间线，例如：

```text
3月：你更多在纠结自己的事，习惯一个人扛
5月：你开始关心身边人的感受，愿意主动表达了
8月：你能同时关注自己和他人，不再觉得倾诉是负担
```

### 与事件记忆的关系

事件是具体的"发生了什么"，观察是抽象的"你在怎么变化"。
两者互补：事件提供证据，观察提炼趋势。

------

## 8.2 历史机制（已移除）

### 八维权重（已移除）

> v0.3 移除。用 ±0.02 微调八个浮点数来量化成长是虚假的精确。
> 改为定性观察，用自然语言描述变化。

### Reinforcement（已移除）

> v2 移除。逐轮强化导致波动过大。

### Compensation（已移除）

> v0.3 移除。八维权重移除后，临时补偿机制不再适用。

# 9. Reflection Engine（成长观察模块）✅ 待重构

从每周调权重，改为每周生成观察。

------

## Reflection 触发条件

### 每周 ✅ 已实现（需重构）

生成：

- Weekly Reflection（读取周报+事件+日记+已有观察 → 生成新的观察描述）
- 事件归并（检查新事件，自动归入已有项目或发现新项目）

------

## Reflection 内容

AI 分析本周对话，生成观察：

- 用户在关注什么？有什么变化？
- 用户是否在重复某种模式？
- 和之前的观察相比，有什么新的趋势？
- 用自然语言描述，不量化

**观察原则**：

- 基于事实（有事件/对话支撑），不臆测
- 描述变化趋势，不做评判
- 可以不确定（"你好像开始..."）
- 不夸大（没有明显变化就不写）

------

## Reflection 告别数字

> **v0.3 核心变更**：移除了 ±0.02 的硬限制和八维权重系统。
> 人的成长不是数字能表达的。用文字诚实地描述变化，比用数字假装精确更好。

# 12. 项目档案与成长线 ✅ 待实现（v0.3 新增）

- 周汇总时自动检查：新事件能否归入已有项目，或发现新项目

### 数据结构

```sql
CREATE TABLE projects (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,          -- 项目标题，如"考研"、"转行前端"

  updated_at DATETIME

# 13. 心跳机制 ✅ 已实现（v0.2 新增，v0.3 重新定位）

后台异步循环，让 AI 具备时间感知能力。

## 13.1 心跳循环

每 30 分钟执行一次：

1. **遗忘曲线维护**：衰减所有事件的 strength，清理已遗忘事件
2. **时间感知更新**：记录用户最后活跃时间，感知空闲时长
3. **主动消息检查**：空闲太久时自然地打招呼

## 13.2 主动消息触发条件

| 条件 | 触因 | 消息风格 |
|------|------|---------|
| 空闲 > 24h | long_idle | "好久没聊了，最近怎么样？" |
| 空闲 > 3天 | very_long_idle | "最近还好吗？有啥想聊的随时来" |

消息 1-2 句，自然随意，不矫情，不提及 AI 身份。

> **v0.3 变更**：移除了情感驱动的主动消息（lonely/excited），只保留时间驱动。
> 心跳的核心价值是"时间感知"而非"情感陪伴"。

前端通过轮询 `/api/heartbeat` 获取主动消息。

  content TEXT NOT NULL,
  emotions TEXT,          -- JSON array: ["sadness", "anger"]
  importance REAL,        -- 0.0 ~ 1.0
  event_type TEXT,        -- "conflict" | "milestone" | "emotion" | "decision"
  strength REAL,          -- 遗忘曲线强度 0.0 ~ 1.0
  stability REAL,         -- 记忆稳定天数
  recall_count INTEGER,   -- 被回忆的次数
  last_recalled_at TEXT,  -- 上次被回忆的时间
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

> **主题关联**：事件通过 `topic_links` 关联表与主题关联（一个事件可关联多个主题），而非直接在 events 表上存 topic_id。

```sql

  first_mentioned TEXT,         -- 首次出现日期

  mention_count INTEGER,        -- 被提及次数
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 事件/观察关联主题
CREATE TABLE topic_links (
  topic_id TEXT NOT NULL,
  item_id TEXT NOT NULL,        -- 事件或观察的 ID

);
```

主题不是预定义的，是 AI 在抽取事件/观察时动态识别和归类的。
同一主题积累了 ≥2 条跨时间段的记录后，可生成主题对比。

# 18. 产品壁垒

真正壁垒不是：

- 大模型

而是：

------

## （1）对话即日记的体验

别人需要用户主动写日记。

你只需要聊天。

------

## （2）日记串联能力

别人保存的是孤立的日记。

你保存的是被串联的人生。

------

## （3）长期记忆积累

用户用得越久，AI 越了解用户，日记质量越高，对比越有价值。

------

# 19. 风险与伦理

------

## 风险 1：记忆隐私 ⚠️ 部分实现

必须：

- ✅ 用户可删除记忆（DELETE 端点已实现）
- ❌ 用户可关闭长期记忆（无全局开关）
- ❌ 敏感内容本地加密（当前明文存储）

日记内容可能非常私密，隐私保护是底线。

**待实现**：
- 记忆开关（用户可选择关闭长期记忆）
- 本地数据加密（至少对事件记忆中的敏感字段加密）

------

## 风险 2：AI 过度解读

AI 在生成观察和对比时，可能过度解读用户的经历。

**待实现**：
- 观察内容应基于事实，标注不确定处
- 用户可以编辑/删除 AI 生成的观察和对比

------

# 21. 最终产品体验

用户随手聊了三年。

AI 不只是记录了这件事，而是帮你记住了：
你走过的路、你犹豫过的选择、你当时的心情。

这些碎片，在 AI 的沉淀下，变成了一笔宝贵的财富。
